열거형(enum)은 어떤 값이 정해진 여러 경우 중 하나라는 사실을 타입으로 표현하는 방법이다. 구조체가 관련 있는 필드를 하나의 타입으로 묶는다면, 열거형은 가능한 상태나 형태를 하나의 타입 안에 배리언트(variant)로 나열한다.

```rust
enum IpAddrKind {
    V4,
    V6,
}

let four = IpAddrKind::V4;
let six = IpAddrKind::V6;
```

`IpAddrKind::V4`와 `IpAddrKind::V6`는 서로 다른 배리언트지만 둘 다 `IpAddrKind` 타입의 값이다. 그래서 IP 주소 종류를 받는 함수는 배리언트별 타입을 따로 받을 필요 없이 하나의 타입만 받으면 된다.

```rust
fn route(ip_kind: IpAddrKind) {
    // ...
}

route(IpAddrKind::V4);
route(IpAddrKind::V6);
```

배리언트 이름 앞에 `IpAddrKind::`를 붙이는 이유는 열거형 이름이 네임스페이스 역할을 하기 때문이다. 같은 이름의 배리언트가 다른 열거형 안에 있을 수 있으므로, 어떤 타입의 배리언트인지 함께 적는다.

## 배리언트에 데이터 담기

열거형은 단순히 이름만 나열하는 데서 끝나지 않는다. 각 배리언트가 데이터를 직접 가질 수 있다.

```rust
enum IpAddr {
    V4(String),
    V6(String),
}

let home = IpAddr::V4(String::from("127.0.0.1"));
let loopback = IpAddr::V6(String::from("::1"));
```

이 형태에서는 `IpAddr::V4`와 `IpAddr::V6`가 각각 `String`을 받아 `IpAddr` 값을 만드는 생성자 함수처럼 동작한다. 구조체로 `kind`와 `address`를 따로 묶지 않아도, "이 값은 IPv4 주소다"와 "그 주소 데이터는 이것이다"가 한 타입 안에 붙어 있다.

배리언트마다 데이터의 모양이 달라도 된다.

```rust
enum IpAddr {
    V4(u8, u8, u8, u8),
    V6(String),
}

let home = IpAddr::V4(127, 0, 0, 1);
let loopback = IpAddr::V6(String::from("::1"));
```

이 점이 구조체와 다른 중요한 지점이다. 구조체는 모든 인스턴스가 같은 필드 모양을 가진다. 반면 열거형은 같은 타입 안에서도 배리언트별로 다른 데이터를 담을 수 있다.

```rust
enum Message {
    Quit,
    Move { x: i32, y: i32 },
    Write(String),
    ChangeColor(i32, i32, i32),
}
```

`Quit`은 데이터가 없고, `Move`는 이름 있는 필드가 있으며, `Write`는 문자열 하나를 담고, `ChangeColor`는 정수 세 개를 담는다. 그래도 네 값은 모두 `Message` 타입이다. 그래서 메시지를 처리하는 함수는 여러 구조체 타입을 따로 받는 대신 `Message` 하나를 받을 수 있다.

## 열거형에도 메서드를 붙일 수 있다

구조체처럼 열거형에도 `impl` 블록으로 메서드를 정의할 수 있다.

```rust
enum Message {
    Quit,
    Move { x: i32, y: i32 },
    Write(String),
    ChangeColor(i32, i32, i32),
}

impl Message {
    fn call(&self) {
        // self는 호출된 Message 값을 가리킨다.
    }
}

let message = Message::Write(String::from("hello"));
message.call();
```

메서드 안에서 어떤 배리언트인지에 따라 다른 동작을 하려면 보통 `match`와 함께 쓴다.

## `Option<T>`

Rust에는 일반적인 의미의 null이 없다. 대신 값이 있을 수도 있고 없을 수도 있는 상태를 `Option<T>` 열거형으로 표현한다.

```rust
enum Option<T> {
    None,
    Some(T),
}
```

`Some(T)`는 값이 있음을 나타내고, `None`은 값이 없음을 나타낸다. `Option<T>`와 그 배리언트인 `Some`, `None`은 프렐루드(prelude)에 포함되어 있어서 보통 `Option::Some`처럼 길게 쓰지 않아도 된다.

```rust
let some_number = Some(5);
let some_char = Some('e');
let absent_number: Option<i32> = None;
```

`None`만 적으면 어떤 타입의 값이 없다는 뜻인지 컴파일러가 추론하기 어렵다. 그래서 위 예제처럼 `Option<i32>`를 명시해야 하는 경우가 있다.

`Option<T>`의 핵심은 `T`와 다른 타입이라는 점이다.

```rust
let x: i8 = 5;
let y: Option<i8> = Some(5);

let sum = x + y; // 컴파일되지 않는다.
```

`Option<i8>`는 "있을 수도 있고 없을 수도 있는 i8"이지, 그냥 `i8`이 아니다. 그래서 값을 실제로 사용하려면 `Some`과 `None`을 먼저 처리해야 한다. 이 강제가 null을 값처럼 잘못 쓰는 실수를 줄인다.

## match

`match`는 값을 패턴(pattern)에 맞춰 갈라 처리하는 표현식이다. 열거형과 함께 쓰면 배리언트별 처리를 명확히 적을 수 있다.

```rust
enum Coin {
    Penny,
    Nickel,
    Dime,
    Quarter,
}

fn value_in_cents(coin: Coin) -> u8 {
    match coin {
        Coin::Penny => 1,
        Coin::Nickel => 5,
        Coin::Dime => 10,
        Coin::Quarter => 25,
    }
}
```

`match` 뒤에는 비교할 값이 오고, 각 갈래(arm)는 `패턴 => 표현식` 형태로 쓴다. 위 함수에서 각 갈래의 결과값은 전체 `match` 표현식의 결과값이 된다.

갈래 안에서 여러 줄을 실행해야 하면 블록을 쓴다.

```rust
fn value_in_cents(coin: Coin) -> u8 {
    match coin {
        Coin::Penny => {
            println!("Lucky penny!");
            1
        }
        Coin::Nickel => 5,
        Coin::Dime => 10,
        Coin::Quarter => 25,
    }
}
```

## 배리언트 내부 값 꺼내기

배리언트가 데이터를 담고 있으면 `match` 패턴에서 그 값을 변수로 바인딩할 수 있다.

```rust
#[derive(Debug)]
enum UsState {
    Alaska,
}

enum Coin {
    Penny,
    Nickel,
    Dime,
    Quarter(UsState),
}

fn value_in_cents(coin: Coin) -> u8 {
    match coin {
        Coin::Penny => 1,
        Coin::Nickel => 5,
        Coin::Dime => 10,
        Coin::Quarter(state) => {
            println!("State quarter from {:?}!", state);
            25
        }
    }
}
```

`Coin::Quarter(state)`는 `Quarter` 배리언트와 매칭되면서 내부의 `UsState` 값을 `state`라는 이름으로 꺼낸다. 이 방식은 `Option<T>`를 다룰 때도 똑같이 쓰인다.

```rust
fn plus_one(x: Option<i32>) -> Option<i32> {
    match x {
        None => None,
        Some(i) => Some(i + 1),
    }
}
```

`Some(i)`는 `Some` 안의 값을 `i`로 꺼낸다. `None`에는 꺼낼 값이 없으므로 그대로 `None`을 돌려준다.

## match는 모든 경우를 다뤄야 한다

`match`는 철저함(exhaustiveness)을 검사한다. 가능한 경우 중 하나라도 빠지면 컴파일되지 않는다.

```rust
fn plus_one(x: Option<i32>) -> Option<i32> {
    match x {
        Some(i) => Some(i + 1),
    }
}
```

이 코드는 `None`을 처리하지 않으므로 컴파일되지 않는다. 이 성질 때문에 `Option<T>`를 쓸 때 값이 없는 경우를 잊기 어렵다.

모든 값을 하나씩 나열하기 어렵거나, 나머지 경우를 한꺼번에 처리하고 싶을 때는 포괄 패턴(catch-all pattern)을 쓴다.

```rust
let dice_roll = 9;

match dice_roll {
    3 => add_fancy_hat(),
    7 => remove_fancy_hat(),
    other => move_player(other),
}
```

`other`는 앞의 패턴에 맞지 않은 값을 모두 받아서 변수로 바인딩한다. 이 값이 필요 없으면 `_`를 쓴다.

```rust
match dice_roll {
    3 => add_fancy_hat(),
    7 => remove_fancy_hat(),
    _ => reroll(),
}
```

아무 일도 하지 않겠다는 뜻을 명시하고 싶으면 유닛 값 `()`을 둘 수 있다.

```rust
match dice_roll {
    3 => add_fancy_hat(),
    7 => remove_fancy_hat(),
    _ => (),
}
```

포괄 패턴은 앞에서부터 순서대로 검사된다는 점 때문에 보통 마지막에 둔다. 먼저 나오면 뒤의 갈래가 실행될 기회가 사라진다.

## if let

`if let`은 하나의 패턴만 관심 있고 나머지는 무시해도 될 때 쓰는 간결한 문법이다.

```rust
let config_max = Some(3u8);

if let Some(max) = config_max {
    println!("The maximum is configured to be {}", max);
}
```

위 코드는 아래 `match`와 같은 의미다.

```rust
let config_max = Some(3u8);

match config_max {
    Some(max) => println!("The maximum is configured to be {}", max),
    _ => (),
}
```

대신 `if let`은 `match`가 주는 철저한 검사를 포기한다. `Some`일 때만 처리하고 `None`은 무시해도 되는 상황이라면 읽기 좋지만, 모든 경우를 의식적으로 다뤄야 한다면 `match`가 더 낫다.

`else`도 붙일 수 있다.

```rust
let coin = Coin::Penny;
let mut count = 0;

if let Coin::Quarter(state) = coin {
    println!("State quarter from {:?}!", state);
} else {
    count += 1;
}
```

이 형태는 한 가지 패턴만 특별히 처리하고, 나머지는 같은 방식으로 처리할 때 적합하다.
