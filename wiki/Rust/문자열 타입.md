Rust에서 문자열은 하나의 타입으로 끝나지 않는다. 소유하는 문자열과 빌려 보는 문자열이 분리되어 있고, 둘 사이 변환에는 복사와 할당 비용이 들어갈 수 있다. 여기에 UTF-8까지 들어오면 "첫 번째 글자" 같은 표현도 단순하지 않다.

## `String`과 `&str`

- `String`: 힙 버퍼를 소유하는 문자열
- `&str`: 문자열 데이터에 대한 읽기 전용 뷰

구조체 안에 값을 오래 보관하거나 함수 밖으로 소유권을 넘겨야 하면 `String`이 필요하다. 이미 존재하는 문자열 데이터를 읽기만 한다면 `&str`로 충분하다.

문자열 리터럴은 `String`이 아니라 `&str`이다.

```rust
let a = "hello";
let b = String::from("hello");
```

`a`는 바이너리 안의 불변 문자열을 가리키는 슬라이스이고, `b`는 힙에 새 버퍼를 가진 소유 문자열이다.

Rust 핵심 언어에 있는 문자열 타입은 `str`이고, 보통은 참조 형태인 `&str`로 다룬다. `String`은 표준 라이브러리가 제공하는 타입이다. 커질 수 있고, 수정할 수 있고, 값을 소유하며, UTF-8로 인코딩된 문자열이다.

## `String`은 바이트 벡터에 UTF-8 의미를 얹은 타입이다

`String`은 내부적으로 바이트를 저장하는 컬렉션에 가깝다. 다만 아무 바이트나 담는 것이 아니라 유효한 UTF-8이라는 보장을 유지한다. 그래서 `Vec<T>`처럼 새 값을 만들고, 늘리고, 내용을 붙이는 API를 가지면서도 문자열 특유의 제약이 따라온다.

빈 문자열은 `String::new()`로 만들 수 있다.

```rust
let mut s = String::new();
```

초깃값이 있으면 `String::from`이나 `to_string`을 쓴다.

```rust
let s1 = String::from("initial contents");
let s2 = "initial contents".to_string();
```

둘 다 문자열 리터럴에서 소유하는 `String`을 만든다. 이 경우에는 스타일과 가독성의 차이에 가깝다.

## 문자열을 늘릴 때는 무엇을 빌리고 무엇을 소유하는지 본다

```rust
let mut s = String::from("hello");
s.push_str(", world");
```

`push_str`은 `&str`을 받아 현재 `String` 뒤에 붙인다. 인자의 소유권을 가져가지 않으므로, 붙인 뒤에도 원래 문자열 슬라이스를 계속 사용할 수 있다.

```rust
let mut s1 = String::from("foo");
let s2 = "bar";

s1.push_str(s2);
println!("{s2}");
```

문자 하나를 붙일 때는 `push`를 쓴다.

```rust
let mut s = String::from("lo");
s.push('l');
```

`+` 연산자로도 문자열을 합칠 수 있지만, 이때 왼쪽 `String`은 이동된다.

```rust
let s1 = String::from("Hello, ");
let s2 = String::from("world!");

let s3 = s1 + &s2;
```

이후 `s1`은 사용할 수 없다. `+`가 내부적으로 `add(self, s: &str)` 형태로 동작하기 때문이다. 왼쪽 값은 소비하고, 오른쪽 값은 참조로 받는다. `&s2`의 타입은 `&String`이지만, 역참조 강제 변환(deref coercion)을 통해 `&str`처럼 쓰일 수 있다.

여러 문자열을 조합할 때는 `format!`이 더 읽기 쉽다.

```rust
let s1 = String::from("tic");
let s2 = String::from("tac");
let s3 = String::from("toe");

let s = format!("{s1}-{s2}-{s3}");
```

`format!`은 인자의 소유권을 가져가지 않고 새 `String`을 만든다.

## 변환 비용

`&str`을 `String`으로 바꾸면 새 버퍼를 할당하고 내용을 복사해야 한다.

```rust
let owned = value.to_string();
```

이 작업이 루프 안에서 반복되면 비용이 눈에 띄게 커질 수 있다. 반대로 `String`에서 `&str`을 얻는 것은 이미 가진 버퍼를 빌려 보는 것이므로 상대적으로 가볍다.

```rust
let s = "Artwork".to_string();
print_name(s.as_str());
```

## `as_`와 `to_`

문자열 관련 메서드는 접두사만 봐도 방향이 보이는 경우가 많다.

- `as_`: 기존 값을 다른 뷰로 본다
- `to_`: 새 소유 데이터를 만든다

`as_str()`는 `String`을 `&str`로 빌려 보고, `to_string()`은 새 `String`을 만든다.

## 시그니처에서 무엇을 받을까

함수가 입력을 읽기만 한다면 `&str`이 더 일반적이다. 호출자가 `String`을 갖고 있어도 `&str`로 넘길 수 있고, 문자열 리터럴도 바로 받을 수 있다.

```rust
fn print_admiration(name: &str) {
    println!("{}", name);
}
```

반대로 함수가 입력 문자열을 직접 소유해야 하거나 구조체 안에 저장해야 하면 `String`이 필요하다.

즉 "읽기만 하는가, 보관하거나 수정하는가"가 시그니처 선택 기준이 된다.

## `String`은 숫자 하나로 인덱싱하지 않는다

다른 언어의 감각으로는 아래 코드가 첫 번째 문자를 꺼낼 것처럼 보인다.

```rust
let s = String::from("hello");
let h = s[0];
```

Rust에서는 컴파일되지 않는다. `String`을 숫자 하나로 인덱싱했을 때 무엇을 돌려줘야 하는지 명확하지 않기 때문이다.

`String`은 UTF-8 바이트를 저장한다. ASCII 문자열에서는 한 글자가 1바이트인 경우가 많지만, 모든 문자가 그렇지는 않다. 예를 들어 키릴 문자나 한글 같은 문자는 여러 바이트로 저장된다. 숫자 인덱스 0이 "첫 번째 바이트"인지, "첫 번째 유니코드 스칼라 값"인지, 사용자가 눈으로 보는 "첫 번째 글자"인지가 달라진다.

또 하나의 이유는 성능이다. 인덱싱은 보통 상수 시간(O(1))을 기대받지만, UTF-8 문자열에서 n번째 문자를 찾으려면 앞에서부터 문자 경계를 확인해야 할 수 있다.

## 문자열을 보는 세 관점

UTF-8 문자열은 적어도 세 관점으로 볼 수 있다.

- 바이트(byte): 실제 저장 단위다.
- 유니코드 스칼라 값(Unicode scalar value): Rust의 `char`에 가깝다.
- 문자소 클러스터(grapheme cluster): 사용자가 하나의 글자로 느끼는 단위에 가장 가깝다.

예를 들어 힌디어 "नमस्ते"는 바이트로 보면 더 길고, `char`로 보면 여섯 개의 스칼라 값이지만, 사용자가 보는 글자 단위로는 네 묶음에 가깝다. 발음 구별 부호가 별도 스칼라 값으로 들어가기 때문이다.

Rust 표준 라이브러리는 바이트와 `char` 반복을 명시적으로 제공한다.

```rust
for c in "Зд".chars() {
    println!("{c}");
}
```

```rust
for b in "Зд".bytes() {
    println!("{b}");
}
```

문자소 클러스터 단위 처리가 필요하면 표준 라이브러리만으로는 부족할 수 있고, 별도 크레이트를 검토해야 한다.

## 범위 슬라이스는 바이트 경계를 직접 다루는 일이다

숫자 하나로 인덱싱하는 것은 막혀 있지만, 범위를 이용해 문자열 슬라이스를 만들 수는 있다.

```rust
let hello = "Здравствуйте";
let s = &hello[0..4];
```

이 코드는 첫 4바이트에 해당하는 `&str`을 만든다. 이 문자열에서는 앞의 두 글자가 각각 2바이트이므로 결과는 "Зд"다.

하지만 문자 경계가 아닌 곳을 자르면 런타임에 패닉이 발생한다.

```rust
let hello = "Здравствуйте";
let s = &hello[0..1];
```

이런 코드는 첫 글자의 중간 바이트를 자르려 하므로 유효한 `&str`을 만들 수 없다. 그래서 문자열 슬라이싱은 "문자 번호"가 아니라 "UTF-8 문자 경계를 지키는 바이트 범위"를 다루는 작업으로 봐야 한다.

## `String::with_capacity`

길이를 미리 아는 큰 문자열이라면 버퍼를 먼저 잡을 수 있다.

```rust
let mut x = String::with_capacity(10_000_000);
for _ in 0..10_000_000 {
    x.push('.');
}
```

용량을 먼저 확보하면 `push` 과정에서 재할당 횟수를 줄일 수 있다.

## 정리

- `String`은 소유하고 수정 가능한 UTF-8 문자열이다.
- `&str`은 이미 존재하는 UTF-8 문자열 데이터의 읽기 전용 뷰다.
- 읽기만 하는 함수는 보통 `&str`을 받는 편이 더 넓게 쓸 수 있다.
- `push_str`은 문자열 슬라이스를 빌리고, `push`는 문자 하나를 붙인다.
- `+`는 왼쪽 `String`을 소비하고, `format!`은 소유권을 가져가지 않는 조합에 유용하다.
- Rust는 `String`의 숫자 인덱싱을 허용하지 않는다.
- 문자 단위 작업은 `chars`, 바이트 단위 작업은 `bytes`처럼 관점을 명시해야 한다.
- 범위 슬라이스는 바이트 경계를 직접 다루므로 UTF-8 문자 경계를 지켜야 한다.
