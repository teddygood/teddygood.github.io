Rust에서 문자열은 하나의 타입으로 끝나지 않는다. 소유하는 문자열과 빌려 보는 문자열이 분리되어 있고, 둘 사이 변환에는 복사와 할당 비용이 들어갈 수 있다. 이 차이는 함수 시그니처와 메모리 비용에 바로 드러난다.

## `String`과 `&str`

- `String`: 힙 버퍼를 소유하는 문자열
- `&str`: 문자열 데이터에 대한 읽기 전용 뷰

구조체 안에 값을 오래 보관하거나 함수 밖으로 소유권을 넘겨야 하면 `String`이 필요하다. 이미 존재하는 문자열 데이터를 읽기만 한다면 `&str`로 충분하다.

문자열 리터럴은 `String` 이 아니라 `&str` 이다.

```rust
let a = "hello";
let b = String::from("hello");
```

`a` 는 바이너리 안의 불변 문자열을 가리키는 슬라이스이고, `b` 는 힙에 새 버퍼를 가진 소유 문자열이다.

## 왜 둘이 나뉘나

`String`은 버퍼를 직접 관리한다. 길이가 달라질 수 있고, 새 문자열을 만들거나 이어 붙일 수 있다. 반면 `&str`은 이미 있는 버퍼의 일부를 가리키기만 한다. 그래서 `&str`은 뷰이고, `String`은 소유 타입이다.

```rust
let mut s = String::from("hello");
s.push_str(", world");
```

이처럼 문자열 내용을 바꾸려면 `String` 이 필요하다.

## 변환 비용

`&str`을 `String`으로 바꾸면 새 버퍼를 할당하고 내용을 복사해야 한다.

```rust
let owned = value.to_string();
```

이 작업이 루프 안에서 반복되면 비용이 눈에 띄게 커질 수 있다. 반대로 `String`에서 `&str`을 얻는 것은 이미 가진 버퍼를 빌려 보는 것이므로 상대적으로 가볍다.

```rust
let s = "Artwork".to_string();
print_name(s.as_str());
```

## `as_`와 `to_`

문자열 관련 메서드는 접두사만 봐도 방향이 보이는 경우가 많다.

- `as_`: 기존 값을 다른 뷰로 본다
- `to_`: 새 소유 데이터를 만든다

`as_str()`는 `String`을 `&str`로 빌려 보고, `to_string()`은 새 `String`을 만든다.

## 시그니처에서 무엇을 받을까

함수가 입력을 읽기만 한다면 `&str`이 더 일반적이다. 호출자가 `String`을 갖고 있어도 `&str`로 넘길 수 있고, 문자열 리터럴도 바로 받을 수 있다.

```rust
fn print_admiration(name: &str) {
    println!("{}", name);
}
```

반대로 함수가 입력 문자열을 직접 소유해야 하거나 구조체 안에 저장해야 하면 `String`이 필요하다.

즉 "읽기만 하는가, 보관하거나 수정하는가"가 시그니처 선택 기준이 된다.

## `String::with_capacity`

길이를 미리 아는 큰 문자열이라면 버퍼를 먼저 잡을 수 있다.

```rust
let mut x = String::with_capacity(10_000_000);
for _ in 0..10_000_000 {
    x.push('.');
}
```

용량을 먼저 확보하면 `push` 과정에서 재할당 횟수를 줄일 수 있다.
