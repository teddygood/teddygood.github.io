해시맵(hash map)은 키로 값을 찾는 컬렉션이다. Rust 표준 라이브러리에서는 `HashMap<K, V>` 타입으로 제공하며, 키 타입 `K`와 값 타입 `V`를 정해 둔다. 벡터가 "몇 번째 값인가"로 접근한다면, 해시맵은 "어떤 키에 연결된 값인가"로 접근한다.

```rust
use std::collections::HashMap;

let mut scores = HashMap::new();

scores.insert(String::from("Blue"), 10);
scores.insert(String::from("Yellow"), 50);
```

`HashMap`은 프렐루드(prelude)에 자동으로 들어오지 않으므로 보통 `use std::collections::HashMap;`을 먼저 쓴다. 벡터의 `vec!`처럼 자주 쓰는 기본 매크로도 따로 없다.

## 키 타입과 값 타입은 각각 하나로 정해진다

해시맵도 벡터처럼 동질적인 컬렉션이다. 모든 키는 같은 타입이어야 하고, 모든 값도 같은 타입이어야 한다.

```rust
let mut scores = HashMap::new();

scores.insert(String::from("Blue"), 10);
scores.insert(String::from("Yellow"), 50);
```

이 코드는 `HashMap<String, i32>`로 추론된다. 이후 같은 맵에 다른 타입의 키나 값을 섞어 넣을 수 없다.

## 조회는 `Option<&V>`로 돌아온다

해시맵은 키가 없을 수 있다. 그래서 `get`은 값 자체가 아니라 `Option<&V>`를 반환한다.

```rust
let team_name = String::from("Blue");
let score = scores.get(&team_name).copied().unwrap_or(0);
```

여기서 `get(&team_name)`의 결과는 `Option<&i32>`다. `copied()`로 `Option<i32>`로 바꾸고, 키가 없으면 `unwrap_or(0)`으로 기본값을 쓴다.

반복할 때는 키와 값의 쌍을 얻는다.

```rust
for (team, score) in &scores {
    println!("{team}: {score}");
}
```

해시맵의 순회 순서는 고정된 삽입 순서가 아니다. 출력이나 테스트에서 순서를 전제하면 안 된다.

## 소유권 있는 키와 값은 맵으로 이동한다

`i32`처럼 `Copy`인 값은 삽입될 때 복사된다. `String`처럼 소유권이 있는 값은 해시맵으로 이동한다.

```rust
let field_name = String::from("Favorite color");
let field_value = String::from("Blue");

let mut map = HashMap::new();
map.insert(field_name, field_value);
```

이후 `field_name`과 `field_value`는 더 이상 사용할 수 없다. 해시맵이 그 값을 소유하기 때문이다.

참조를 값으로 넣을 수도 있지만, 이 경우 참조가 가리키는 원본 값은 해시맵이 쓰이는 동안 계속 유효해야 한다. 이 문제는 결국 라이프타임(lifetime) 규칙으로 이어진다.

## 같은 키에 다시 넣으면 덮어쓴다

해시맵에서 하나의 키는 하나의 값만 가진다. 같은 키로 다시 `insert`하면 기존 값은 새 값으로 바뀐다.

```rust
let mut scores = HashMap::new();

scores.insert(String::from("Blue"), 10);
scores.insert(String::from("Blue"), 25);

println!("{scores:?}");
```

결과적으로 `Blue`에는 `25`가 연결된다.

## 없을 때만 넣으려면 `entry`를 쓴다

키가 이미 있으면 기존 값을 유지하고, 없을 때만 기본값을 넣는 패턴은 자주 나온다. 이때 `entry`와 `or_insert`를 쓴다.

```rust
let mut scores = HashMap::new();
scores.insert(String::from("Blue"), 10);

scores.entry(String::from("Yellow")).or_insert(50);
scores.entry(String::from("Blue")).or_insert(50);
```

`Yellow`는 없으므로 `50`이 들어가고, `Blue`는 이미 있으므로 기존 `10`이 유지된다. 이 API는 직접 `contains_key`로 확인한 뒤 다시 삽입하는 코드보다 대여 규칙과 잘 맞는다.

## 기존 값을 바탕으로 업데이트할 수 있다

단어 등장 횟수처럼 "없으면 0으로 시작하고, 있으면 값을 증가"시키는 경우에도 `entry`가 자연스럽다.

```rust
use std::collections::HashMap;

let text = "hello world wonderful world";
let mut counts = HashMap::new();

for word in text.split_whitespace() {
    let count = counts.entry(word).or_insert(0);
    *count += 1;
}
```

`or_insert`는 해당 값의 가변 참조 `&mut V`를 반환한다. 그래서 `*count += 1`처럼 역참조해서 실제 값을 바꾼다. 이 가변 참조는 루프 반복이 끝날 때 스코프 밖으로 나가므로 다음 반복에서 다시 안전하게 맵을 빌릴 수 있다.

## 기본 해시 함수는 보안 쪽에 무게가 있다

Rust 표준 `HashMap`은 기본 해시 함수로 SipHash 계열을 사용한다. 목적은 해시 테이블을 노리는 서비스 거부 공격(DoS attack)에 대한 저항성을 제공하는 것이다. 가장 빠른 선택은 아닐 수 있지만, 일반적인 기본값으로는 보안 쪽 균형을 택한 것이다.

성능 측정 결과 해시 함수가 병목이라면 다른 해시어(hasher)를 지정할 수 있다. 이때 해시어는 `BuildHasher` 트레이트를 구현한 타입이다. 보통은 직접 구현하기보다 목적에 맞는 크레이트를 검토하는 편이 현실적이다.
