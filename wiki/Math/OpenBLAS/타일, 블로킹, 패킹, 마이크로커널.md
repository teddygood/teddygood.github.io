GEMM 최적화에서 **타일(tile)**, **블로킹(blocking)**, **패킹(packing)**, **마이크로커널(microkernel)**은 따로 떨어진 용어가 아니다. 큰 행렬 곱셈을 cache, TLB, register, SIMD pipeline에 맞춰 여러 층의 작은 계산으로 나누는 같은 구조를 서로 다른 각도에서 부르는 말이다.

행렬 곱셈은 수학적으로는 단순하다.

$$
C_{ij} = \sum_{p=0}^{k-1} A_{ip}B_{pj}
$$

하지만 실제 성능은 이 수식을 몇 번 계산하느냐만으로 결정되지 않는다. 같은 값을 memory hierarchy 안에서 얼마나 오래 재사용하느냐가 성능을 좌우한다. 그래서 GEMM 구현을 읽을 때는 수식 자체보다 **데이터가 어느 memory layer에 있고, 어떤 모양으로 읽히며, 가장 안쪽 계산 루프가 어떤 tile을 갱신하는가**를 더 중요하게 보게 된다.

## Tile

타일은 큰 행렬이나 반복 공간을 하드웨어가 잘 다룰 수 있는 작은 조각으로 자른 단위다. 수학적으로 새로운 연산이 생기는 것은 아니다. 같은 `C := A B + C` 계산을 cache, TLB, register, SIMD, thread 배치에 맞게 나눠 실행하기 위해 만든 작업 단위다.

가장 중요한 점은 tile이 고정된 크기나 고정된 층을 뜻하지 않는다는 점이다. 문맥에 따라 L1/L2/L3 cache에 맞춘 큰 block일 수도 있고, microkernel이 register에 들고 있는 아주 작은 output 조각일 수도 있다.

예를 들어 microkernel이 `m_r x n_r = 4 x 8`을 쓴다면, 가장 안쪽에서는 `C`의 4행 8열짜리 작은 조각을 register에 올려 둔다. 그 상태에서 `k_c` 방향으로 반복하며 `A`의 `4 x k_c` micro-panel과 `B`의 `k_c x 8` micro-panel을 곱해 누적한다. 이 4 x 8 출력 조각이 register tile 또는 microtile이다.

바깥쪽에는 더 큰 tile도 있다. `A`의 `m_c x k_c` block, `B`의 `k_c x n_c` panel, `C`의 `m_c x n_c` 영역처럼 cache와 TLB에 맞춰 잡는 조각이다. 같은 단어 tile이 나오더라도 "어느 memory layer에서 재사용하려는 조각인가"를 먼저 봐야 한다.

## Blocking

블로킹은 큰 행렬 계산을 tile 또는 block 단위로 나누는 전략이다. 행렬 전체를 한 번에 다루면 cache에 들어온 데이터가 금방 밀려나기 쉽다. 그래서 큰 문제를 조각으로 나누고, 한 조각 안에서 같은 `A`와 `B` 데이터를 여러 번 다시 쓰게 만든다.

GEMM 구현에서는 여러 층의 블로킹이 동시에 등장한다. 큰 `C` matrix를 `n_c`, `m_c`, `k_c` 같은 cache block size로 나누고, 마지막에는 `m_r x n_r` 같은 register tile을 갱신한다. 여기서 `m_c`, `k_c`, `n_c`는 cache와 TLB에 맞춘 크기이고, `m_r`, `n_r`는 microkernel이 register에서 한 번에 유지할 출력 tile 크기다.

블로킹은 크게 두 관점으로 나눠 볼 수 있다.

- Cache blocking은 L1/L2/L3 cache에 맞춰 큰 matrix를 block이나 panel로 나누는 일이다.
- Register blocking은 출력의 작은 tile을 register에 올려 두고, 여러 `k` step 동안 partial sum을 누적하는 일이다.

핵심은 "작게 자른다"가 아니라 "작게 자른 뒤 같은 데이터를 더 오래 재사용한다"다.

## Tiling과 Blocking

타일링(tiling)과 블로킹은 GEMM 문맥에서 거의 같은 뜻으로 쓰인다. 큰 반복 공간이나 matrix를 작은 조각으로 나눠, 한 조각 안에서 데이터 재사용을 늘리는 변환이다.

다만 어감은 조금 다르다. Blocking은 cache blocking, register blocking처럼 최적화 전략을 말할 때 자주 쓰이고, tile은 그 결과 생긴 개별 조각을 가리킬 때 자주 쓰인다. 컴파일러 쪽에서는 loop tiling이라고 부르고, BLAS 쪽에서는 block size나 cache block이라는 표현이 더 자주 나온다.

컴파일러 문맥의 loop tiling은 반복 공간을 나누는 루프 변환(loop transformation) 자체가 중심이다. 그쪽 설명은 [Loop Tiling](../../Compiler/Loop%20Tiling.md)에 따로 정리해뒀다.

tile은 정사각형일 필요가 없다. GEMM에서는 대부분 직사각형이다. `m_c`, `k_c`, `n_c`, `m_r`, `n_r` 같은 파라미터가 서로 다른 이유도 각 차원이 cache, register, SIMD width, packing format에 받는 제약이 다르기 때문이다.

## Packing

패킹은 계산 전에 matrix block이나 panel을 계산하기 좋은 memory layout으로 다시 복사하는 일이다. 원본 matrix는 leading dimension, stride, transpose, submatrix view 때문에 실제 메모리에서 값이 연속적이지 않을 수 있다. 이 상태 그대로 microkernel에 넣으면 load pattern이 복잡해지고, SIMD와 prefetch도 불리해진다.

그래서 GEMM 구현은 필요한 `A` block과 `B` panel을 packed buffer로 옮긴 뒤 계산한다. 논문에서는 보통 `A~`, `B~`처럼 표현한다. 이 buffer는 microkernel이 기대하는 순서로 값이 놓여 있어야 한다.

패킹의 의미는 단순한 복사가 아니다. 복사 비용은 분명 overhead지만, 그 뒤의 큰 계산에서 다음 이득을 얻기 위한 준비 단계다.

- 연속 메모리 접근을 만든다.
- SIMD load와 broadcast를 단순하게 만든다.
- TLB entry 사용을 줄인다.
- 같은 packed panel을 여러 output block 계산에서 재사용한다.
- Transpose나 conjugation 같은 operand 변형을 계산 루프 안이 아니라 packing 단계에서 처리할 수 있다.

작은 matrix에서는 packing overhead가 계산량보다 크게 보일 수 있다. 그래서 packing은 "항상 좋은 복사"가 아니라, 복사 비용을 계산에서 회수할 수 있을 때 의미가 있다.

## Microkernel

마이크로커널은 가장 안쪽에서 작은 출력 tile 하나를 실제로 계산하는 핵심 코드다. 여기서 kernel은 운영체제 kernel이 아니라 "핵심 계산 루프"라는 뜻이다.

GEMM microkernel은 보통 `m_r x n_r` 크기의 `C` tile을 register에 두고, `k_c` 방향으로 반복하면서 multiply-add를 누적한다. 구조는 대략 아래처럼 생각할 수 있다.

```text
for p in 0..k_c:
    load A micro-panel
    load or broadcast B micro-panel
    C registers += A * B
store C registers
```

microkernel은 대상 아키텍처에 강하게 맞춰진다. vector width, register 수, load 명령, FMA throughput이 다르면 같은 GEMM이라도 `m_r x n_r` tile 모양과 packed layout이 달라진다.

OpenBLAS, BLIS, MKL 같은 라이브러리에서 CPU별 커널 코드가 따로 있는 이유가 여기에 있다. GEMM 전체 구조는 비슷해도, 성능 차이는 microkernel과 그 주변 packing 코드에서 크게 난다.

## Tile 계층

GEMM 구현을 읽을 때는 tile을 아래처럼 층별로 보면 덜 헷갈린다.

| 층 | 자주 보이는 이름 | 크기 예시 | 주된 목적 |
|---|---|---|---|
| Cache / TLB | block, panel, cache block | `m_c x k_c`, `k_c x n_c`, `m_c x n_c` | cache와 TLB 안에서 재사용할 만큼 크게 잡는다 |
| Packed buffer | packed block, packed panel | packed `A_c`, packed `B_c` | unit-stride 접근, aligned load, 예측 가능한 cache set mapping을 만든다 |
| Microkernel 입력 | micro-panel | `m_r x k_c`, `k_c x n_r` | microkernel이 연속적으로 읽을 수 있게 만든다 |
| Register | register tile, microtile | `m_r x n_r` | `C` partial sum을 register에 유지하고 `k_c`번 누적한다 |
| Thread scheduling | loop tile, work chunk | `ic`, `jr`, `ir` loop 조각 | 여러 thread가 어떤 `C` 영역과 packed data를 공유할지 정한다 |

이 표에서 가장 안쪽의 microtile은 출력 `C`의 조각이고, micro-panel은 입력 `A` 또는 `B`의 조각이다. 이름이 비슷하지만 역할이 다르다. Microtile은 계산 결과를 누적하는 곳이고, micro-panel은 그 계산에 들어가는 데이터를 공급하는 곳이다.

## 전체 구조

세 용어는 순서대로 이어진다. 먼저 blocking으로 큰 GEMM을 cache와 register에 맞는 조각으로 나눈다. 그다음 packing으로 `A`, `B` 조각을 microkernel이 읽기 좋은 packed buffer에 옮긴다. 마지막으로 microkernel이 그 packed data를 읽어서 `C`의 작은 output tile을 register에서 갱신한다.

셋 중 하나만 좋아서는 부족하다. 블로킹만 있고 패킹이 없으면 microkernel이 읽는 데이터 모양이 복잡해진다. 패킹만 있고 microkernel이 약하면 SIMD와 register를 충분히 못 쓴다. microkernel이 빨라도 바깥 block size가 맞지 않으면 packing 비용이나 cache miss가 커진다.

큰 GEMM -> cache block / panel로 분할 -> `A`, `B` 일부를 packed buffer로 복사 -> microkernel이 `C`의 작은 tile을 register에서 갱신 -> 다음 tile로 이동.

## 관련 용어

용어는 표로만 짧게 정리한다.

| 용어                | 의미                                                               |
| ----------------- | ---------------------------------------------------------------- |
| Block             | Matrix를 자른 조각을 넓게 가리키는 말이다.                                      |
| Tile              | Block과 거의 같지만, 특히 loop tiling이나 output 조각을 말할 때 자주 보인다.          |
| Panel             | Matrix의 넓거나 긴 조각이다. GEMM에서는 `A` block, `B` panel 같은 식으로 자주 나온다.  |
| Microtile         | Microkernel이 직접 갱신하는 `C`의 `m_r x n_r` 출력 조각이다.                   |
| Micro-panel       | Microkernel이 직접 소비하는 더 작은 packed input panel이다.                  |
| Leading dimension | 2차원 matrix에서 다음 column이나 row로 넘어갈 때 실제 memory에서 건너뛰는 간격이다.       |
| Stride            | 다음 원소까지의 memory 간격이다. View나 transpose가 있으면 unit stride가 아닐 수 있다. |
| Macro-kernel      | BLIS 문맥에서 microkernel 주변 loop를 portable code로 감싼 층이다.            |
| Inner-kernel      | GotoBLAS 문맥에서 microkernel보다 큰 안쪽 계산 단위를 가리키는 표현으로 자주 나온다.        |
