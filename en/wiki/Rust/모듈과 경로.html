모듈과 경로

코드가 커지면 먼저 어려워지는 것은 파일 개수가 아니라 이름이 어디까지 보이는지다. 함수 몇 개를 다른 파일로 옮겼는데 갑자기 cannot find function이나 private function 에러가 나는 이유도 여기에 있다. Rust에서 모듈은 단순한 파일 분리가 아니라 네임스페이스와 가시성 규칙을 같이 만드는 장치다.

패키지, 크레이트, 모듈​

Rust 프로젝트를 읽을 때는 먼저 세 단어를 분리해두는 편이 좋다.

  • 패키지(package): Cargo.toml을 가진 Cargo 단위다. 빌드, 테스트, 공유의 기준이 된다.
  • 크레이트(crate): Rust 컴파일러가 한 번에 컴파일하는 코드 단위다. 바이너리 크레이트와 라이브러리 크레이트가 있다.
  • 모듈(module): 크레이트 안에서 이름과 공개 범위를 나누는 구조다.

크레이트 루트(crate root)는 컴파일러가 읽기 시작하는 파일이다. 바이너리 크레이트의 기본 루트는 src/main.rs이고, 라이브러리 크레이트의 기본 루트는 src/lib.rs다.

my-project
├── Cargo.toml
└── src
    └── main.rs

이 구조에서는 src/main.rs가 패키지 이름과 같은 바이너리 크레이트의 루트가 된다. src/lib.rs가 있으면 패키지 이름과 같은 라이브러리 크레이트도 생긴다. src/bin/ 아래에 파일을 두면 각각이 별도의 바이너리 크레이트가 된다.

한 패키지에는 여러 바이너리 크레이트가 있을 수 있지만, 라이브러리 크레이트는 하나만 둘 수 있다. 보통 실행 진입점은 src/main.rs에 얇게 두고, 재사용 가능한 핵심 로직은 src/lib.rs에 두면 바이너리도 라이브러리의 공개 API를 쓰는 사용자처럼 동작하게 된다.

모듈 트리​

모듈은 크레이트 안에 트리 구조를 만든다.

mod front_of_house {
    mod hosting {
        fn add_to_waitlist() {}
        fn seat_at_table() {}
    }

    mod serving {
        fn take_order() {}
        fn serve_order() {}
        fn take_payment() {}
    }
}

위 코드는 아래와 같은 모듈 트리를 만든다.

crate
└── front_of_house
    ├── hosting
    │   ├── add_to_waitlist
    │   └── seat_at_table
    └── serving
        ├── take_order
        ├── serve_order
        └── take_payment

최상위에는 암묵적으로 crate가 있고, 그 아래에 부모 모듈과 자식 모듈이 생긴다. hosting과 serving은 같은 부모를 가진 형제 모듈이다. 이 트리를 기준으로 경로와 공개 여부가 결정된다.

모듈을 만들면 이름이 다른 자리에 생긴다​

처음에는 함수가 모두 같은 파일에 있어서 아무 경로 없이 바로 호출된다.

fn main() {
    let name = get_name();
    hello(&name);
    goodbye(&name);
}

여기서 get_name, hello, goodbye를 input, output 모듈로 옮기면 main은 더 이상 그 이름들을 바로 볼 수 없다.

mod input {
    fn get_name() -> String { ... }
}

mod output {
    fn hello(name: &str) { ... }
    fn goodbye(name: &str) { ... }
}

이 상태에서 main은 get_name()을 찾지 못한다. 함수가 사라진 것이 아니라, input::get_name과 output::hello처럼 다른 네임스페이스 안으로 들어갔기 때문이다.

그래서 첫 단계는 경로를 가져오는 것이다.

use input::get_name;
use output::{goodbye, hello};

이제 이름은 찾지만, 바로 다음 에러가 나온다. 함수가 기본적으로 private이기 때문이다.

기본값은 private다​

Rust에서는 함수, 구조체, 열거형, 모듈이 기본적으로 private이다. use는 이름을 현재 스코프로 가져올 뿐이고, private 규칙 자체를 바꾸지는 않는다.

그래서 모듈 밖에서 써야 하는 함수에는 pub를 붙여야 한다.

mod input {
    pub fn get_name() -> String { ... }
}

mod output {
    pub fn hello(name: &str) { ... }
    pub fn goodbye(name: &str) { ... }
}

모듈화할 때 use와 pub가 같이 나오는 이유가 여기 있다.

  • mod는 이름을 새 네임스페이스 안으로 넣는다.
  • use는 그 경로를 현재 스코프로 끌어온다.
  • pub는 그 이름을 바깥에서 보이게 만든다.

파일로 나누려면 mod 선언이 먼저 필요하다​

모듈을 인라인 블록으로 둘 수도 있지만, 보통은 파일로 분리한다.

use input::get_name;
use output::{goodbye, hello};

mod input;
mod output;

여기서 mod input;은 단순한 스타일 문제가 아니다. Rust 컴파일러에게 input.rs 파일을 모듈로 읽으라고 알려 주는 선언이다. 파일이 있다고 자동으로 모듈이 생기지 않는다. mod가 없으면 컴파일러는 그 파일을 찾지 않는다.

crate::와 super::는 기준점을 바꾼다​

경로는 크게 두 종류로 읽으면 된다.

  • 상대 경로: 현재 모듈을 기준으로 찾는다.
  • 절대 경로: 크레이트 루트를 기준으로 찾는다.

절대 경로의 시작점은 crate::다.

use crate::day_kind::DayKind;

이 형태가 필요한 이유는 모듈 안에서 루트 모듈의 이름을 바로 찾을 수 없기 때문이다. 예를 들어 output.rs 안에서 use day_kind::DayKind;라고 쓰면 현재 모듈 기준으로 day_kind를 찾으려 하기 때문에 실패한다. day_kind가 크레이트 루트 바로 아래에 있으니 crate::day_kind::DayKind라고 적어야 한다.

super::는 한 단계 위 부모 모듈을 뜻한다.

pub mod shop {
    fn buy() -> super::Treat {
        super::Treat::IceCream
    }
}

이런 식으로 부모 모듈의 타입을 가리킬 수 있다. 다만 코드가 이동하면 기준점도 함께 바뀌므로, 수정 지점을 줄이려면 crate::...로 시작하는 절대 경로와 use를 함께 두는 쪽이 안정적이다.

use는 긴 경로를 한 번만 적기 위한 장치다​

절대 경로를 그대로 매번 쓰면 시그니처가 길어진다.

fn regret(treat: crate::treats::ConsumedTreat) {
    println!("That was a mistake");
}

이럴 때는 use로 한 번 끌어오고, 나머지 코드는 짧게 쓰면 된다.

use crate::treats::ConsumedTreat;

fn regret(treat: ConsumedTreat) {
    println!("That was a mistake");
}

이 방식의 장점은 코드 이동에 강하다는 점이다. 모듈 구조를 바꾸더라도 수정 지점이 use문 근처로 모이기 쉽다.

use는 해당 스코프 안에만 단축 경로를 만든다. 루트 모듈에 use crate::front_of_house::hosting;를 적었다고 해서 자식 모듈 안에서도 자동으로 hosting을 볼 수 있는 것은 아니다. 자식 모듈에서 쓰려면 그 안에 다시 use를 두거나, 부모 기준 경로인 super::hosting을 써야 한다.

함수를 가져올 때는 보통 함수 이름까지 직접 가져오기보다 부모 모듈을 가져온다.

use crate::front_of_house::hosting;

pub fn eat_at_restaurant() {
    hosting::add_to_waitlist();
}

use crate::front_of_house::hosting::add_to_waitlist;로 함수 자체를 가져올 수도 있지만, 호출부에서 add_to_waitlist()만 보이면 이 함수가 로컬에 정의된 것처럼 읽힐 수 있다. 부모 모듈을 남겨두면 경로는 짧아지면서도 어디에서 온 함수인지 드러난다.

반대로 구조체, 열거형, 트레이트 같은 타입은 전체 경로를 가져오는 방식이 보통 자연스럽다.

use std::collections::HashMap;

let mut scores = HashMap::new();

같은 이름의 타입을 둘 이상 가져와야 하면 부모 모듈을 남기거나 as로 이름을 바꾼다.

use std::fmt;
use std::io;

fn render() -> fmt::Result {
    Ok(())
}

fn read() -> io::Result<()> {
    Ok(())
}
use std::fmt::Result;
use std::io::Result as IoResult;

fn render() -> Result {
    Ok(())
}

fn read() -> IoResult<()> {
    Ok(())
}

pub use는 가져온 이름을 다시 외부로 내보낸다. 내부 모듈 구조와 사용자가 보게 될 API 구조를 다르게 만들 때 쓴다.

mod front_of_house {
    pub mod hosting {
        pub fn add_to_waitlist() {}
    }
}

pub use crate::front_of_house::hosting;

이렇게 하면 외부 코드는 내부 구조를 전부 알 필요 없이 restaurant::hosting::add_to_waitlist()처럼 사용할 수 있다. 라이브러리를 만들 때 pub use는 공개 API를 정리하는 도구가 된다.

외부 패키지도 결국 경로와 use로 가져온다. Cargo.toml에 의존성을 추가하면 Cargo가 패키지를 내려받고, 코드에서는 크레이트 이름으로 시작하는 경로를 쓴다.

rand = "0.8.5"
use rand::Rng;

let secret_number = rand::thread_rng().gen_range(1..=100);

표준 라이브러리 std도 경로 관점에서는 외부 크레이트처럼 다룬다. Rust에 포함되어 있으므로 Cargo.toml에 추가하지 않을 뿐, 코드에서 쓰려면 use std::... 경로를 적는다.

여러 use가 같은 앞부분을 공유하면 중첩 경로로 줄일 수 있다.

use std::{cmp::Ordering, io};
use std::io::{self, Write};

글롭(glob) 연산자인 *는 해당 경로의 공개 아이템을 모두 가져온다.

use std::collections::*;

편하지만 이름이 어디에서 왔는지 흐려지기 쉽다. 테스트 모듈이나 프렐루드(prelude) 패턴처럼 의도가 분명한 곳에서 제한적으로 쓰는 편이 읽기 쉽다.

pub(crate)는 밖에는 숨기고 안에서는 연다​

pub는 범위가 너무 넓고, private은 범위가 너무 좁을 때가 있다. 크레이트 안에서는 여러 모듈이 같이 써야 하지만, 외부 사용자에게는 보여 주지 않을 함수가 대표적이다. 이럴 때 pub(crate)를 쓴다.

mod forest {
    pub(crate) fn enter_area(area: &str) {
        match area {
            "tree cover" => println!("It's getting darker..."),
            "witches coven" => println!("It's getting spookier..."),
            "walking path" => println!("It's getting easier to walk..."),
            x => panic!("Unexpected area: {}", x),
        }
    }
}

이 함수는 현재 크레이트 안에서는 어디서나 쓸 수 있지만, 다른 크레이트에는 공개되지 않는다. 외부 API와 내부 유틸리티의 범위를 분리할 때 쓰인다.

자식은 부모의 private을 볼 수 있지만, 반대는 아니다​

Rust 모듈에서 자주 놓치는 규칙이 하나 더 있다. 자식 모듈은 부모 모듈의 private 아이템을 볼 수 있다. 하지만 부모나 형제 모듈은 자식 모듈의 private 아이템을 직접 볼 수 없다.

즉, 이런 방향은 된다.

fn function() {}

mod nested {
    fn function() {
        crate::function();
    }
}

하지만 이런 방향은 안 된다.

fn function() {
    nested::function();
}

mod nested {
    fn function() {}
}

그래서 pub(crate)가 붙은 함수가 private 모듈 안에 있어도, 같은 크레이트의 다른 내부 코드에서 접근 가능한 경우가 생긴다. 가시성은 "파일이 다르다"보다 "모듈 트리에서 어느 방향인가"에 더 크게 좌우된다.

공개 구조체와 공개 열거형은 다르게 열린다​

구조체에 pub를 붙이면 구조체 이름은 공개되지만, 필드는 여전히 기본적으로 private이다. 필드별로 공개 여부를 따로 정해야 한다.

mod back_of_house {
    pub struct Breakfast {
        pub toast: String,
        seasonal_fruit: String,
    }

    impl Breakfast {
        pub fn summer(toast: &str) -> Breakfast {
            Breakfast {
                toast: String::from(toast),
                seasonal_fruit: String::from("peaches"),
            }
        }
    }
}

pub fn eat_at_restaurant() {
    let mut meal = back_of_house::Breakfast::summer("Rye");
    meal.toast = String::from("Wheat");
}

Breakfast는 공개 타입이고 toast는 공개 필드이므로 바깥에서 수정할 수 있다. 하지만 seasonal_fruit는 private이라 직접 읽거나 바꿀 수 없다. private 필드가 있는 구조체는 바깥에서 리터럴로 직접 만들 수 없으므로, 보통 공개 생성 함수가 필요하다.

열거형은 다르다. 열거형 자체를 공개하면 배리언트도 공개된다.

mod back_of_house {
    pub enum Appetizer {
        Soup,
        Salad,
    }
}

pub fn eat_at_restaurant() {
    let order1 = back_of_house::Appetizer::Soup;
    let order2 = back_of_house::Appetizer::Salad;
}

열거형은 배리언트를 통해 값을 만들고 구분하는 타입이라, 타입만 공개하고 배리언트를 전부 숨기면 쓸 수 있는 여지가 매우 작다. 그래서 구조체 필드는 따로 열어야 하지만, 공개 열거형의 배리언트는 함께 열린다.

파일로 나누는 규칙​

파일 분리는 모듈 트리를 바꾸는 작업이 아니다. 이미 있는 모듈의 본문을 다른 파일로 옮기는 작업이다.

// src/lib.rs
mod front_of_house;

pub use crate::front_of_house::hosting;

pub fn eat_at_restaurant() {
    hosting::add_to_waitlist();
}

mod front_of_house;는 컴파일러에게 front_of_house 모듈의 본문을 다른 위치에서 찾으라고 알려준다. 관용적인 위치는 src/front_of_house.rs다.

// src/front_of_house.rs
pub mod hosting {
    pub fn add_to_waitlist() {}
}

자식 모듈도 파일로 분리하려면 부모 모듈 파일에 선언만 남긴다.

// src/front_of_house.rs
pub mod hosting;

그리고 자식 모듈의 본문은 부모 모듈 이름의 디렉터리 아래에 둔다.

// src/front_of_house/hosting.rs
pub fn add_to_waitlist() {}

중요한 점은 mod가 include처럼 매번 파일을 붙여 넣는 명령이 아니라는 것이다. 모듈 트리에서 한 번 선언하면, 다른 파일에서는 그 모듈의 경로를 사용해 접근한다.

Rust는 예전 스타일인 mod.rs 경로도 지원한다.

src/front_of_house.rs
src/front_of_house/mod.rs

src/front_of_house/hosting.rs
src/front_of_house/hosting/mod.rs

같은 모듈에 대해 두 스타일을 동시에 쓰면 컴파일 에러가 난다. 서로 다른 모듈에 두 스타일을 섞어 쓰는 것은 가능하지만, 에디터에 mod.rs가 여러 개 열리는 구조는 읽는 사람을 헷갈리게 만들 수 있다.

모듈을 나눌 때 자주 보는 에러는 결국 세 종류로 정리된다.

  1. 이름을 못 찾는다: 경로를 잘못 썼다.
  2. 이름은 찾는데 private이다: pub나 pub(crate)가 빠졌다.
  3. 경로가 너무 길고 자주 깨진다: use로 기준점을 먼저 정리해야 한다.
